## 一、系统全景

### 1.1 模块清单

这套系统由 4 层 11 个模块组成。每个模块的定位和职责如下。

**数据层（1 个模块）**

DailyCache 包装 DataFetcher，同一 API 同参数当日只调一次。16 次 API 调用降到 8 次。你不需要关心它，它在 final_system.py 初始化时自动生效。

**感知层（5 个 Skill）**

SentimentSkill 是短期体温计。只看今天的涨跌比、涨跌停和 1-3 天的北向资金。输出一个 -2 到 +2 的温度值，+2 表示市场极度亢奋（事实描述，不含方向判断）。Sentiment 不告诉你该买还是该卖。

MacroSkill 是中期水位表。看 20 天的 SHIBOR 利率、北向资金趋势、两融余额趋势。回答"现在市场有没有钱"这个问题。流动性充裕不代表该买入，但流动性枯竭时一定不该加仓。

MarketEnvironment v6.1 是环境评分。综合多项指标给出 0-100 分的环境评分。这是整个系统最硬的开关——低于 60 分时 TradeSignalScanner 自动跳过买入扫描，辩论引擎也会被拦截。你不需要判断"今天该不该交易"，这个分数替你判断。

SectorRotationSkill 是板块绝对排名。用 5 日、20 日、60 日绝对涨幅的百分位排名给 31 个申万行业打分。输出 Top N 强势板块和轮动信号。

SectorStageFilter 是板块相对排名 + 个股形态。在 SectorRotation 基础上增加三件事：第一，用相对强度（vs 沪深300/创业板指）替代绝对涨幅，牛市中所有板块都涨的情况下能区分谁更强。第二，Bottom 5 板块拉入黑名单，禁止买入且限期退出。第三，在 Top 板块内搜索周线大底（60-500 天横盘）+ 日线收紧（10 天窄幅缩量 + EMA 骑乘）的个股。

**决策层（3 个模块）**

StockPipeline 是选股流水线。CANSLIM 基本面评分 + 缩量整理形态 + 板块排名加分 → 输出候选池。这是 SectorStageFilter 的下游消费者——StageFilter 筛完的候选再经 CANSLIM 打分。

TradeSignalScanner 是扣扳机模块。其他所有模块回答"这只股票好不好"，这个模块回答"今天要不要动手"。它扫描候选池的今日行情，检测 4 种买入触发（阶梯突破、EMA 突破、放量阳线、缺口突破）和 5 种卖出触发（止损、趋势破位、板块退出、止盈分批、时间退出），计算固定风险仓位，并标注紧急程度。

DebateEngine 是辩论引擎。对 TradeSignalScanner 产生的买入信号启动多空辩论。看多分析师、看空分析师、风控三视角（逆向/顺势/均衡）各出一段论证，主席裁决。只在 needs_debate=True 且环境评分 ≥ 60 时触发。止损信号不过辩论，直接执行。

**反馈层（1 个模块）**

StrategyReviewerV2 是复盘引擎。每日自动记录信号和辩论决策，回填 5/10 日实际收益。每周五计算信号记分卡（按类型统计胜率）和辩论正确率。可选接入 LLM 做深度复盘。产出结构化参数变更提案，必须人工审批才能生效，30 天后自动回滚。

**基础设施（Brain、Router、CronEngine 等）**

Router 接收自然语言问题，路由到 1-2 个 Skill 执行，整合结果生成回答。Brain 是长期记忆，记录每次操作的上下文和教训。CronEngine 按时间表触发盘前/盘后自动任务。Scratchpad 记录每次 Skill 调用的输入输出。

## 二、日常操作流程

### 2.1 周日晚上：定战略方向（30 分钟）

这是一周中最重要的 30 分钟。不看个股，只看板块。

**操作：** 手动触发 SectorStageFilter

```python
filter = SectorStageFilter()
result = filter.analyze(verbose=True)
print(result.to_brief())
```

**你需要做的决策：**

第一，确认本周的"作战区域"——RS 排名前 5 的板块。只在这些板块里找机会，其他板块的机会再好也不追。

第二，检查黑名单。如果你有持仓落在 Bottom 5 板块里，本周优先退出。不是因为它一定会跌，而是资金在离开这些板块，你留在里面是逆流而行。

第三，看轮动预警。如果上周的领涨板块出现"动能衰退"或"更低高点"警告，收紧该板块持仓的止损。领涨不会永远持续。

第四，记录 Stage 候选。有哪些个股处于"周线大底 + 日线收紧"状态？这些是下周的重点观察对象，但现在不做买入决策。

**不需要辩论引擎。** 这一步只是确定方向。

### 2.2 交易日盘前 8:30（5 分钟）

CronEngine 自动执行，你只需要看结果。

**自动触发的任务：**

```
MarketEnvironment.evaluate()  → 环境评分
SentimentSkill.analyze()      → 情绪温度
MacroSkill.analyze()          → 流动性
RiskControlSkill.analyze()    → 持仓体检
```

**你看到的核心数据（一屏摘要）：**

```
[环境] 72/100 → 正常交易日
[情绪] +0.8 贪婪 | 仓位建议: ≤50%(逆向:偏热→谨慎)
[宏观] +0.3 中性偏松 | SHIBOR 平稳, 北向净流入
[风控] 总仓位 36%/50%上限 | 止损预警: 比亚迪(-6.8%)
```

**你的判断只有三种：**

环境 ≥ 70：正常交易日。可以执行买入计划，仓位按正常标准。

环境 60-69：谨慎日。只减仓不加仓。如果 TradeSignalScanner 产生了买入信号，仓位减半。

环境 < 60：停手日。不开新仓，不问辩论引擎。唯一要做的是检查止损。

这个判断不需要花时间思考，环境评分就是答案。

### 2.3 交易日盘后 16:00（15 分钟）

这是系统最核心的工作时段。

**自动触发的任务：**

```python
# CronEngine 盘后任务
signal_report = scanner.scan(
    watchlist=pipeline_candidates,      # 候选池
    holdings=current_positions,          # 当前持仓
    env_score=env_score,                 # 环境评分
    blacklisted_sectors=blacklist,       # 黑名单
    total_capital=total_capital,
    current_total_position=current_pos,
)
```

**卖出信号处理（先看卖出）：**

止损信号（urgency=立即）：不讨论，不犹豫，下一个交易日开盘执行。这是整个系统最铁的纪律。如果你发现自己在想"再等等看"，说明你在用感情替代纪律。

趋势破位（urgency=今日）：EMA20 下方放量收盘。和止损一样执行，但可以在心里记住是哪只股票，作为复盘材料。

板块退出（urgency=3 日内）：所在板块进入黑名单。不急但必须做，给自己 3 个交易日的窗口找合适的退出点。

止盈分批（urgency=今日）：盈利达到 1R 卖 1/3、2R 再卖 1/3、3R 剩余用移动止损。这是让利润奔跑的机制。

时间退出（urgency=观察）：持仓超过 20 天浮盈不到 5%。这只股票占用了资金但没有表现，考虑换掉。

**买入信号处理：**

环境 < 60 时：不会产生买入信号（被门控拦截），你不需要做任何事。

环境 ≥ 60 时：对每个买入信号，系统已经计算好入场价、止损价、目标价、盈亏比、仓位。你需要做的是把信号送进辩论引擎。

```python
# 对每个买入信号
for signal in signal_report.buy_signals:
    question = f"{signal.name}触发{signal.signal_type}，是否执行买入？"
    debate_result = debate_engine.run(question, context=all_briefs)
```

辩论引擎会输出：看多论证、看空论证、风控三视角、主席裁决（买入/观望/放弃 + 置信度）。

你的最终决策只有三种：

裁决为"买入"且置信度 ≥ 70%：按计算仓位执行。
裁决为"买入"但置信度 60-69%：仓位减半。
裁决为"观望"或置信度 < 60%：不执行。

每天最多执行 2-3 个买入决策。超过这个数量说明不是在精选而是在扫货。

**记录：**

```python
# 盘后最后一步
reviewer.log_signal_outcomes(signal_report.buy_signals + signal_report.sell_signals)
if debate_result:
    reviewer.log_debate_outcome(debate_result.to_dict())
reviewer.update_outcomes(fetcher=fetcher)
```

### 2.4 周五盘后：复盘（20 分钟）

**自动执行：**

```python
review = reviewer.run_weekly_review(
    current_params=SKILL_PARAMS,
    fetcher=fetcher,
)
```

**你需要看的：**

信号记分卡：哪种信号类型胜率最高、哪种最低。如果"阶梯突破"胜率 70% 而"缺口突破"只有 30%，下周对缺口突破的信号要格外谨慎。

辩论正确率：辩论说"买"的对了几次，说"观望"的又对了几次。如果辩论说"观望"但实际大涨的比例很高，说明系统可能过于保守。

参数提案：如果样本量够（≥20），系统可能产出 1-2 个参数调整建议。比如"阶梯突破量比阈值从 1.5 提到 1.8，因为过去 30 天 1.5 的门槛让太多假突破通过"。

**审批：** 你决定批准还是驳回。批准后去 config.py 手动改参数。30 天后系统会提醒你是否需要回滚。

## 三、操作纪律

### 3.1 铁律（违反 = 亏钱）

止损触发时不犹豫。不问辩论引擎，不跟任何人讨论，下个交易日开盘卖出。止损的意义不是避免每一次亏损，而是防止一次大亏抹掉几个月的利润。

环境 < 60 时不开新仓。系统设计了环境门控不是为了好看，而是因为回测数据表明低环境评分期间的交易期望值为负。

不买黑名单板块的股票。形态再好、基本面再强，如果所在板块资金在流出，你的胜率会显著下降。

每笔买入必须有明确的止损价和仓位。在你下单之前，必须能说出"止损在 XX 元，亏损 Y%，占总资金 Z%"。如果说不出来，不要买。

### 3.2 灰色地带处理

辩论说"买入"但你心里不舒服：不买。你的风险厌恶感知有时比模型更灵敏。

辩论说"观望"但你很想买：不买。如果你想买的冲动强到想绕过系统，这恰恰说明你在被情绪驱动。

环境 62，辩论说可以买，信号也触发了：仓位减半。灰色地带用半仓应对，不全押也不放弃。

某只股票连续两周出现在候选池但一直没触发：继续等。好的交易就是等到所有条件对齐的那一天。不对齐的时候等待不是浪费时间，是在保护资金。

### 3.3 异常处理

Tushare API 故障导致环境评分无法计算：当作环境 < 60 处理，不交易。宁可错过一天机会，不冒一天风险。

辩论引擎 LLM 调用失败：回退到 TradeSignalScanner 的结果，但仓位自动减半。没有辩论的决策质量更低，用低仓位补偿。

连续 3 天止损触发：RiskControlSkill 的连续止损熔断会自动把仓位上限压到 30%。不需要你手动干预，但你应该停下来想一想是不是市场环境发生了变化。

## 四、系统配置

### 4.1 关键参数及其含义

```python
# config.py
SKILL_PARAMS = {
    "trade_signal": {
        "breakout_vol_ratio": 1.5,    # 阶梯突破量比门槛，越高越严格
        "min_rr_ratio": 3.0,          # 最低盈亏比，低于此不产生信号
        "risk_per_trade": 0.01,       # 单笔最大亏损占总资金1%
        "max_single_position": 0.08,  # 单只最大仓位8%
        "stop_loss_pct": 0.07,        # 默认止损7%
        "time_exit_days": 20,         # 20天无表现退出
    },
    "sector_stage_filter": {
        "top_n": 5,                   # 关注前5板块
        "blacklist_n": 5,             # 拉黑后5板块
        "base_min_days": 60,          # 底部最短60天
        "min_rr_ratio": 3.0,          # Stage过滤盈亏比
    },
    "risk": {
        "max_single_position": 0.10,  # 风控单只上限10%
        "max_sector_exposure": 0.30,  # 单板块上限30%
        "stop_loss_pct": 0.05,        # 风控止损线5%（比信号层更紧）
        "consecutive_stop_limit": 2,  # 连续止损2次触发熔断
    },
}
```

### 4.2 市场状态 → 参数调整建议

牛市确认（环境持续 > 80）：top_n 可扩大到 8，blacklist_n 缩小到 3，min_rr_ratio 可降到 2.5。但不要自动切换，用提案审批机制。

熊市确认（环境持续 < 55）：top_n 缩小到 3，blacklist_n 扩大到 10，min_rr_ratio 升到 4.0，risk_per_trade 降到 0.005。

震荡市（环境 60-75 来回）：保持默认参数。震荡市是最考验纪律的时期，频繁调参数反而有害。

## 五、优化空间

以下是系统当前的真实短板，按优先级排序。

### 5.1 高优先级

**问题 A：持仓数据手动维护。** 当前 RiskControlSkill 读取 positions.json 文件，这个文件需要你手动更新。55 只持仓的日常增删改是繁重的手工活。每次买入/卖出后你必须手动编辑 JSON，容易遗漏、容易出错。

**解决方向：** 建一个 PortfolioManager 模块。它维护 positions.json 的 CRUD 操作，TradeSignalScanner 产生 BUY 信号并被你确认后自动添加持仓记录（含入场价、止损价、买入日期），SELL 信号执行后自动更新或删除。还需要一个简单的持仓导入功能，从券商导出的 CSV 初始化。

**问题 B：回测验证缺失。** trade_signal.py 和 sector_stage_filter.py 的触发逻辑是否真的能赚钱？目前只有单元测试验证逻辑正确性，没有历史数据回测验证盈利能力。backtest_v6.py 存在但它测试的是 MarketEnvironment 的环境门控效果，不是 TradeSignalScanner 的信号质量。

**解决方向：** 写一个 backtest_signal.py，用 2024-2025 的历史数据模拟 TradeSignalScanner 的完整扫描流程：每天扫描候选池 → 检测触发 → 计算仓位 → 记录结果 → 统计胜率/收益/最大回撤。特别要测试"阶梯突破"在 A 股环境下的真实胜率——美股经验不能直接照搬。

**问题 C：CronEngine 任务编排不完整。** 文档描述的盘前/盘后流程是理想化的。实际 CronEngine 中各任务的依赖关系、执行顺序、错误恢复还没有完全串起来。比如 TradeSignalScanner 需要 Pipeline 的候选池、StageFilter 的黑名单、MarketEnvironment 的评分作为输入，这些依赖目前需要手动编排。

**解决方向：** 在 CronEngine 中定义一个 daily_postmarket_pipeline() 方法，按依赖拓扑排序执行所有任务，任一环节失败有明确的降级策略（比如 StageFilter 失败就跳过黑名单过滤，不影响其他环节）。

### 5.2 中优先级

**问题 D：分钟级行情缺失。** TradeSignalScanner 用日线级别数据，只能在盘后扫描。但有些触发信号（比如缺口突破）在盘中 9:30 就已经确认了，等到盘后 16:00 再看已经错过了最佳入场点。

**解决方向：** 这不是单纯的代码问题，而是数据源的问题。Tushare 的分钟级数据有延迟且配额有限。如果要做盘中扫描，需要接入实时行情源（如东方财富 websocket 或券商 API），成本和复杂度会显著上升。建议先跑稳日线级别的系统，确认信号质量达标后再考虑盘中化。目前的替代方案是在盘前设置好条件单：比如 StageFilter 给出候选 A 的突破价是 31.2，你在券商 APP 上设一个"价格 > 31.2 且量比 > 1.5 时提醒"的条件单。

**问题 E：辩论引擎缺乏自我校准。** DebateEngine 的 prompt 是固定的——看多分析师的论证框架、看空分析师的反驳角度、主席的裁决标准都是写死的。不管是牛市还是熊市，辩论的"思维方式"不变。

**解决方向：** 用 StrategyReviewerV2 的辩论正确率数据反馈给 DebateEngine。如果过去 30 天辩论说"买入"的正确率只有 35%，在 prompt 中注入这个事实："注意，近期买入决策正确率偏低，请看空分析师加大质疑力度"。这不是改参数，是改辩论的"气氛"。可以作为 generate_weekly_review() 的附加输出。

**问题 F：没有关联交易检测。** 55 只持仓中可能有高度相关的股票（比如同时持有 3 只半导体股），实际风险暴露远大于表面的板块集中度检查。RiskControlSkill 只看板块标签，不看股票之间的收益率相关性。

**解决方向：** 在 RiskControlSkill 中增加一个 _check_correlation() 方法，计算持仓两两之间近 60 天的收益率相关性。相关性 > 0.7 的股票对标记为"关联持仓"，合并计算风险暴露。这个功能计算量不大但对风控效果提升明显。

### 5.3 低优先级

**问题 G：止盈策略过于机械。** 当前止盈分批是固定的 1R/2R/3R 台阶。但不同市场环境下最优的止盈策略不同——趋势行情中应该更耐心（用移动止损而不是固定台阶），震荡行情中应该更积极（1R 就走大部分）。

**解决方向：** 可以根据环境评分和板块 RS 动态调整止盈台阶。环境 > 80 且板块 RS 仍在上升时，把 1R 止盈的比例从 33% 降到 20%，让利润多跑。环境 60-70 时提高到 50%。这类调整可以通过参数提案机制来实现。

**问题 H：缺少组合级别的收益归因。** 你知道每只股票的盈亏，但不知道"这个月的收益有多少来自板块选择，多少来自个股选择，多少来自仓位管理"。这三者的贡献比例直接决定你应该把精力花在哪里。

**解决方向：** 实现 Brinson 归因模型。把组合收益分解为板块配置效应（你选了哪些板块）、个股选择效应（板块内你选了哪只）、交叉效应。这个功能对长期优化系统方向很有价值，但实现复杂度较高，可以等系统跑稳 3 个月后再做。

**问题 I：LLM 成本优化还有空间。** 当前辩论引擎每次调用 4 次 LLM（看多、看空、风控、裁决）。如果每天 2-3 个买入信号，一个月约 200 次调用。用 DeepSeek V3 成本不高，但如果切到更贵的模型（R1 推理），成本会显著上升。

**解决方向：** 日常辩论用 V3，只在周五复盘会议用 R1。或者实现"快速辩论"模式——置信度很高的信号（R:R > 5 且环境 > 80）只做一次裁决调用，不走完整辩论。

## 六、系统演进路线

按落地先后排序：

```
现在 ─────────────────────────────────────────────────── 3个月后

Phase 1 (本周):
  ✅ 已完成: 全部模块代码就绪
  → 用真实 Tushare 数据端到端跑一遍完整流程
  → 修复接入中发现的数据格式问题

Phase 2 (第1-2周):
  → PortfolioManager: 持仓自动维护
  → CronEngine 完整编排: 盘前/盘后任务链
  → 第一次真实信号输出 (只看不做)

Phase 3 (第3-4周):
  → backtest_signal.py: 用2024-2025数据验证信号胜率
  → 根据回测结果调整参数 (通过提案机制)
  → 小仓位实盘验证 (总资金的10%)

Phase 4 (第2-3个月):
  → 相关性风控
  → 辩论引擎自校准
  → 动态止盈策略
  → 逐步增加实盘仓位

Phase 5 (3个月后):
  → Brinson归因
  → 盘中行情接入
  → 复盘数据足够后的系统性参数优化
```

**最重要的一点：** Phase 2 的"只看不做"阶段不能跳过。让系统跑两周只输出信号但你不执行，用这两周的信号结果来验证系统是否靠谱。如果两周的模拟信号胜率低于 40%，必须先回去查问题，不要带着未验证的系统上实盘。
